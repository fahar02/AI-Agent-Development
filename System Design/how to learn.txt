STEP 1: Understand What System Design Is
-----------------------------------------
- System design = the process of defining the architecture, components,
  modules, interfaces, and data flow for a system to satisfy requirements.
- Two types:
    a) High-Level Design (HLD) → Architecture, components, APIs, databases
    b) Low-Level Design (LLD) → Class diagrams, DB schema, code-level design
- Why it matters: Every real-world product (Google, Netflix, Uber) is built on
  solid system design decisions.

Resources:
  - Book: "Designing Data-Intensive Applications" by Martin Kleppmann (Bible)
  - YouTube: Gaurav Sen - System Design playlist
  - YouTube: ByteByteGo channel by Alex Xu

STEP 4: Scalability Concepts
------------------------------
  [ ] Vertical Scaling  → Bigger machine (CPU, RAM)
  [ ] Horizontal Scaling → More machines (scale-out)
  [ ] Load Balancing
        - Round Robin, Least Connections, IP Hash strategies
        - Tools: NGINX, HAProxy, AWS ELB
  [ ] Stateless vs Stateful services
  [ ] Session management across servers (sticky sessions, shared cache)

--------------------------------------------------------------------------------

STEP 5: Caching (Very Important!)
-----------------------------------
  [ ] What is caching and WHY it's critical (reduces DB load, low latency)
  [ ] Cache Strategies:
        - Cache-aside (Lazy loading)
        - Write-through
        - Write-back (Write-behind)
        - Read-through
  [ ] Eviction Policies: LRU, LFU, FIFO
  [ ] Cache Invalidation (hardest problem in CS!)
  [ ] Tools: Redis, Memcached
  [ ] CDN as a caching layer for static assets

Practice:
  - Implement a simple LRU cache in Python/Java from scratch

--------------------------------------------------------------------------------

STEP 6: Message Queues & Async Processing
-------------------------------------------
  [ ] Why async? — Decoupling producers and consumers
  [ ] Message Queue vs Event Streaming
        MQ     → RabbitMQ, ActiveMQ (point-to-point)
        Stream → Apache Kafka, AWS Kinesis (pub/sub, replay)
  [ ] Use cases:
        - Email notifications
        - Order processing
        - Log ingestion pipelines
  [ ] Dead Letter Queues (DLQ) — handling failed messages
  [ ] Kafka concepts: topics, partitions, consumer groups, offsets

--------------------------------------------------------------------------------

STEP 7: Database Deep Dive
---------------------------
  [ ] Replication
        - Master-Slave (Primary-Replica) → read replicas
        - Master-Master → both nodes accept writes
  [ ] Sharding (Partitioning)
        - Horizontal sharding — split rows across servers
        - Sharding strategies: Range, Hash, Directory
        - Hotspot problem and consistent hashing
  [ ] Consistent Hashing — used in distributed caches and DBs
        (Visualize: nodes on a ring, data mapped to nearest node)
  [ ] Database Proxies: PgBouncer, ProxySQL
  [ ] NewSQL alternatives: CockroachDB, Spanner

--------------------------------------------------------------------------------

STEP 8: Consistent Hashing (Master This!)
-------------------------------------------
  [ ] Problem: When you add/remove a server, all keys remap → thundering herd
  [ ] Solution: Consistent Hashing places servers and keys on a hash ring
  [ ] Virtual Nodes (vnodes) → distribute load more evenly
  [ ] Used in: Amazon DynamoDB, Apache Cassandra, Memcached

Practice:
  - Implement consistent hashing in code

--------------------------------------------------------------------------------

STEP 9: API Design & Gateway
------------------------------
  [ ] REST API best practices
        - Resource naming (/users/{id})
        - Versioning (/api/v1/)
        - Idempotency (PUT, DELETE)
        - Status codes (200, 201, 400, 401, 403, 404, 500)
  [ ] gRPC — high performance, Protocol Buffers, for internal services
  [ ] GraphQL — flexible queries, single endpoint, used at Facebook/GitHub
  [ ] API Gateway — single entry point, rate limiting, auth, routing
        Tools: Kong, AWS API Gateway, NGINX

STEP 14: Rate Limiting & Throttling
-------------------------------------
  [ ] Why? — prevent abuse, protect backend
  [ ] Algorithms:
        - Fixed Window Counter   → simple but burst-prone
        - Sliding Window Log     → accurate but memory heavy
        - Sliding Window Counter → balance of both
        - Token Bucket           → allows bursting (used by AWS)
        - Leaky Bucket           → smooth output rate
  [ ] Implementation: Redis + Lua scripts (atomic operations)
  [ ] Rate limiting at API Gateway level

STEP 17: System Design Interview Framework
-------------------------------------------
Use this structured approach for every design problem:

  FRAMEWORK (45-minute interview):

  1. CLARIFY REQUIREMENTS (5 min)
     - Functional requirements: what must the system do?
     - Non-functional: scale, availability, consistency, latency
     - Ask: DAU/MAU? Read-heavy or write-heavy? Global or regional?

  2. CAPACITY ESTIMATION (5 min)
     - Traffic: QPS (queries per second)
     - Storage: bytes per record × number of records
     - Bandwidth: data transferred per second
     - Memory: cache sizing

     Quick math cheat sheet:
       1M requests/day  ≈  12 req/sec
       1B requests/day  ≈  11,500 req/sec
       1KB per record × 1B records = 1TB storage

  3. HIGH-LEVEL DESIGN (10 min)
     - Draw the main components: Client → LB → App Servers → DB/Cache
     - Define APIs
     - Choose SQL vs NoSQL

  4. DEEP DIVE (20 min)
     - Bottlenecks? → Add caching, sharding, replication
     - Single points of failure? → Add redundancy
     - Data consistency? → Define consistency model
     - Hot partitions? → Consistent hashing, key design

  5. WRAP UP (5 min)
     - Monitoring & alerting
     - Failure scenarios and mitigation
     - Future improvements

KEY PRINCIPLES TO ALWAYS REMEMBER
================================================================================

  1. "There is no perfect system — only trade-offs."
  2. Always ask WHY before HOW.
  3. Start simple. Add complexity only when needed.
  4. Bottlenecks are usually: DB, network, or single points of failure.
  5. Caching solves 80% of performance problems.
  6. Design for failure — assume everything will break.
  7. Measure first, optimize second. ("Premature optimization is the root of all evil")
  8. Prefer async over sync wherever possible for scalability.
  9. Data >> Opinion. Let load testing guide decisions.
  10. A great system design explanation is 50% communication, 50% knowledge.
