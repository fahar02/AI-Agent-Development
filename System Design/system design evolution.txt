1. THE LAMP STACK
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  LAMP = Linux + Apache + MySQL + PHP
  Think of it like a restaurant:
    Linux   → the building (OS, runs everything)
    Apache  → the waiter (web server, handles requests)
    MySQL   → the menu/storage room (database)
    PHP     → the chef (processes logic, cooks the response)

▸ How it worked (Intermediate)
  - The browser sends a URL request
  - Apache receives it, passes it to PHP
  - PHP queries MySQL and builds an HTML page
  - Apache sends that HTML back to the browser
  - The entire app lived on ONE machine

▸ Why it broke down (Advanced)
  - No separation of concerns: business logic, UI, DB queries all mixed together
  - Scaling = buying a bigger server (vertical scaling only)
  - One bug could crash everything — there was no isolation
  - Thread-based concurrency was expensive and slow under load

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

2. MVC ARCHITECTURE (Model-View-Controller)
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  MVC was the first real attempt to organize code properly:
    Model      → handles data (talks to DB)
    View       → generates what the user sees (HTML)
    Controller → the brain that decides what to do with a request

▸ How it helped (Intermediate)
  - Code became easier to read and maintain
  - Teams could split work: frontend devs on Views, backend on Models
  - Frameworks like Ruby on Rails, Django, Laravel popularized it

▸ Its limits (Advanced)
  - Still ran as a monolith — one deploy, one process
  - Large codebases became "big ball of mud" over time
  - Shared database schema caused tight coupling across features

3. SERVER-SIDE RENDERING (SSR)
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  The server builds a complete HTML page and sends it to the browser.
  The browser just displays it — no JavaScript magic needed.

▸ How it worked (Intermediate)
  - Every click = a full round trip to the server
  - Session state stored in server memory or cookies
  - Great for SEO (Google can read the HTML directly)

▸ Its problems (Advanced)
  - Every page load felt slow (full network round-trip)
  - No real-time features (no live updates, no multiplayer)
  - Server had to render for *every* user on *every* action

4. HORIZONTAL SCALING & LOAD BALANCING
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  Instead of upgrading one server, you add more servers.
  A Load Balancer sits in front and spreads traffic across them.

  [User] → [Load Balancer] → [Server 1]
                           → [Server 2]
                           → [Server 3]

▸ How it worked (Intermediate)
  - Round-robin, least-connections, or IP-hash routing strategies
  - Servers must be STATELESS (no user session stored locally)
  - Session data moved to a shared store (Redis, DB)

▸ Deep concerns (Advanced)
  - Sticky sessions vs. shared session stores (tradeoffs)
  - Health checks: load balancer pings servers, removes dead ones
  - "Split-brain" risk if two servers disagree on state
  - Auto-scaling groups: automatically add/remove servers based on load

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

5. CACHING
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  Instead of hitting the DB every time, store the result in fast memory.
  Think of it as keeping your most-used tools on your desk, not in the basement.

▸ Layers of caching (Intermediate)
  - Browser cache     → stores assets locally on user's machine
  - CDN cache         → stores static files at edge locations worldwide
  - Application cache → Redis/Memcached stores computed DB results
  - DB query cache    → DB itself caches repeated queries

▸ Cache invalidation (Advanced)
  The hardest problem in caching: when do you clear old data?
  Strategies:
    TTL (Time to Live)     → expire after N seconds
    Write-through          → update cache and DB together
    Write-back             → update cache first, DB later (risky)
    Cache-aside (Lazy)     → application checks cache, falls back to DB
  
  Cache stampede: many requests hit DB simultaneously when cache expires
  Solution: probabilistic early expiration, mutex locks

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

6. REDIS
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  Redis = Remote Dictionary Server
  An extremely fast in-memory store. Like RAM, but you can query it.

▸ Common uses (Intermediate)
  - Session storage     → keep user login state across servers
  - Caching             → store expensive DB query results
  - Rate limiting       → count API calls per user per second
  - Pub/Sub messaging   → lightweight event broadcasting
  - Leaderboards        → sorted sets for real-time rankings

▸ Internals (Advanced)
  - Single-threaded event loop (avoids race conditions)
  - Persistence options: RDB (snapshots) vs AOF (append-only log)
  - Redis Cluster: shards data across nodes
  - Sentinel: automatic failover when primary dies

7. CDN (Content Delivery Network)
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  A network of servers placed around the world that serve content
  from the location closest to the user.
  Without CDN: User in India waits for server in USA to respond
  With CDN:    User in India gets content from a server in Mumbai

▸ What gets cached on CDN (Intermediate)
  - Images, videos, CSS, JavaScript files
  - Static HTML pages
  - API responses (with correct cache headers)

▸ Advanced usage (Advanced)
  - Edge-side rendering: run logic at CDN edge (Cloudflare Workers)
  - Cache purging strategies: partial vs. full invalidation
  - Geo-based routing: users routed to nearest region
  - DDoS protection: CDN absorbs attack traffic before origin

8. DATABASE REPLICATION
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  Copy your database to multiple servers.
  One server handles writes, others handle reads.

▸ How it works (Intermediate)
  Primary (master) → accepts all writes
  Replicas (slaves) → copy data from primary, serve reads
  Read-write splitting: application routes queries appropriately

▸ Replication lag & consistency (Advanced)
  - Replica may be milliseconds behind primary (eventual consistency)
  - Strong consistency = slower (synchronous replication)
  - Eventual consistency = faster (asynchronous replication)
  - Multi-primary replication: multiple write nodes, complex conflict resolution
  - Raft & Paxos: consensus algorithms that make replication safe

9. CLOUD PROVIDERS (AWS, Azure, GCP)
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  Rent servers, storage, networking — on demand, pay per minute.
  No buying hardware. No data center maintenance.

▸ Key services (Intermediate)
  - Compute: EC2 (AWS), VMs (Azure), GCE (GCP)
  - Object Storage: S3, Azure Blob, GCS
  - Managed Databases: RDS, Cloud SQL, Aurora
  - DNS & Networking: Route53, VPC, CloudFront

▸ Architecture impact (Advanced)
  - Infrastructure as Code (IaC): Terraform, CloudFormation
  - Immutable infrastructure: never patch servers, replace them
  - Multi-region deployments: active-active for global redundancy
  - Cost optimization: Reserved vs. On-demand vs. Spot instances

10. VIRTUALIZATION & CONTAINERS
────────────────────────────────────────────────────────────────────────────────
▸ Virtualization (Beginner)
  Run many "virtual computers" on one physical machine using a Hypervisor.
  Each VM has its own OS — heavy but isolated.

▸ Docker Containers (Intermediate)
  Containers share the host OS kernel — much lighter than VMs.
  A container packages: code + runtime + libraries + config
  "It works on my machine" → solved by containers

▸ Internals (Advanced)
  - Linux namespaces: isolate processes, network, filesystems
  - cgroups: limit CPU, memory per container
  - Layered filesystems: images built in layers, efficient to cache
  - Container security: non-root users, read-only filesystems, seccomp

11. NoSQL DATABASES
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  Databases that don't use rows and columns (no strict schema).
  Better for flexible, large-scale, distributed data.

▸ Types and use cases (Intermediate)
  Document  → MongoDB     → Store JSON-like docs (user profiles, posts)
  Key-Value → Redis, DynamoDB → Super-fast lookups by key
  Wide-col  → Cassandra   → Write-heavy, time-series, IoT data
  Graph     → Neo4j       → Social networks, recommendation graphs

▸ CAP Theorem (Advanced)
  In a distributed system, you can only guarantee 2 of 3:
    C = Consistency   (all nodes see the same data)
    A = Availability  (every request gets a response)
    P = Partition Tolerance (works despite network splits)
  
  - Cassandra: AP (available + partition tolerant, eventual consistency)
  - MongoDB:   CP (consistent + partition tolerant)
  Trade-offs must be chosen based on business requirements.

12. MICROSERVICES
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  Instead of one big app, break it into many small independent services.
  Each service does ONE job and does it well.
  Example: user-service, payment-service, notification-service

▸ Benefits (Intermediate)
  - Each service can be deployed independently
  - Teams can own and scale their service separately
  - Failure in one service doesn't crash the whole system
  - Different services can use different tech stacks

▸ Real complexity (Advanced)
  - Network calls replace function calls → latency, failures
  - Distributed transactions are hard: 2PC, Saga pattern, outbox pattern
  - Service discovery: how does service A find service B? (Consul, Eureka)
  - Data ownership: each service owns its own DB (no shared DB)
  - Conway's Law: system structure mirrors team structure

13. API GATEWAYS
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  A single front door for all client requests.
  Clients don't talk to services directly — they talk to the gateway.

▸ What it does (Intermediate)
  - Authentication & authorization (JWT validation)
  - Rate limiting (protect services from abuse)
  - Request routing to correct microservice
  - Protocol translation (REST → gRPC)
  - Response aggregation (combine multiple service responses)

▸ Advanced patterns (Advanced)
  - BFF (Backend for Frontend): separate gateway per client type
  - Circuit breaker integration: stop routing to unhealthy services
  - Examples: Kong, AWS API Gateway, Nginx, Envoy

14. SERVICE MESH
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  Infrastructure layer that handles communication BETWEEN services.
  Each service gets a sidecar proxy — it handles all network concerns.

▸ What it gives you (Intermediate)
  - Mutual TLS (mTLS): automatic encryption between services
  - Automatic retries and timeouts
  - Traffic shaping (route 10% of traffic to new version)
  - Observability: metrics and tracing without code changes

▸ Internals (Advanced)
  - Istio / Linkerd: popular service mesh implementations
  - Data plane (Envoy proxy) vs. control plane (Istiod)
  - Sidecar injection: meshes run a proxy container alongside each pod
  - Service mesh adds latency — must be weighed against benefits

15. EVENT-DRIVEN ARCHITECTURE & MESSAGE QUEUES
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  Services communicate by sending events, not direct calls.
  Producer sends event → Broker stores it → Consumer processes it.
  Services are decoupled — they don't know about each other.

▸ Tools and patterns (Intermediate)
  Kafka          → high-throughput, persistent, replayable log
  RabbitMQ       → traditional message broker, flexible routing
  Event sourcing → store state as a sequence of events (not current state)
  CQRS           → separate read model from write model

▸ Deep concepts (Advanced)
  - At-least-once vs. exactly-once delivery guarantees
  - Consumer groups: multiple consumers share load from a topic
  - Dead letter queues: store messages that couldn't be processed
  - Outbox pattern: write event to DB and queue atomically
  - Event schema evolution: avoid breaking consumers when events change

16. KUBERNETES (K8s)
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  A system for automatically deploying, scaling, and managing containers.
  Think of it as an OS for your cluster of servers.

▸ Core concepts (Intermediate)
  Pod         → smallest unit; one or more containers that run together
  Deployment  → manages desired number of pod replicas
  Service     → stable IP/DNS name for a group of pods
  Ingress     → routes external HTTP traffic to services
  ConfigMap   → stores environment config (not secrets)
  Secret      → stores sensitive config (passwords, keys)

▸ Advanced (Advanced)
  - Scheduler: decides which node to place a pod on
  - Horizontal Pod Autoscaler: scales pods based on CPU/custom metrics
  - Operators: custom controllers that manage complex stateful apps
  - Node pools: different VM types for different workloads (GPU nodes)
  - Resource requests vs. limits: critical for reliable scheduling

17. CI/CD PIPELINES
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  CI = Continuous Integration: automatically test every code change
  CD = Continuous Delivery/Deployment: automatically ship to production

▸ Pipeline stages (Intermediate)
  Code push → Build → Unit Tests → Integration Tests → Docker Build
  → Push to Registry → Deploy to Staging → Smoke Tests → Deploy to Prod

▸ Deployment strategies (Advanced)
  - Blue-Green: run two environments, flip traffic instantly, easy rollback
  - Canary: send 5% traffic to new version, monitor, then increase
  - Rolling: replace pods one by one, zero downtime
  - Feature flags: deploy code but hide features behind a toggle
  Tools: GitHub Actions, Jenkins, ArgoCD (GitOps), Tekton

18. SERVERLESS COMPUTING
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  You write a function. Cloud runs it. You pay only when it runs.
  No servers to manage, no idle cost.

▸ How it works (Intermediate)
  - AWS Lambda, Azure Functions, Google Cloud Functions
  - Triggered by HTTP requests, queue messages, DB changes, schedules
  - Cold start: first invocation has extra latency (function boots up)
  - Stateless by design — no local storage between calls

▸ When NOT to use it (Advanced)
  - Long-running tasks (max ~15 mins on Lambda)
  - High-frequency, predictable load (containers are cheaper)
  - Vendor lock-in risk — hard to move between clouds
  - Cold start is unacceptable for latency-sensitive endpoints
  Advanced: Provisioned concurrency, Lambda SnapStart (JVM)

19. EDGE COMPUTING
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  Move computation physically closer to where users are.
  Instead of sending data to a central server, process it at the edge.

▸ Use cases (Intermediate)
  - Serve personalized content from CDN edge (Cloudflare Workers)
  - IoT devices process sensor data locally before sending to cloud
  - Reduce latency for gaming, AR/VR, autonomous vehicles

▸ Architecture (Advanced)
  - Edge nodes are resource-constrained → must be lightweight
  - Consistency challenges: how do edge nodes sync state with origin?
  - WebAssembly (WASM) at the edge: portable, sandboxed, fast
  - Hierarchical edge: device → regional edge → cloud

20. OBSERVABILITY (Monitoring, Logging, Tracing)
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  The ability to understand what's happening inside your system
  from the outside — by looking at data it produces.

▸ The Three Pillars (Intermediate)
  Metrics  → numbers over time (CPU %, error rate, latency p99)
             Tools: Prometheus + Grafana
  Logs     → timestamped text records of events
             Tools: ELK Stack (Elasticsearch, Logstash, Kibana), Loki
  Traces   → follow a single request across all services
             Tools: Jaeger, Zipkin, OpenTelemetry

▸ Advanced observability (Advanced)
  - SLI/SLO/SLA: Service Level Indicators, Objectives, Agreements
  - Error budgets: how much failure you can afford before violating SLO
  - Structured logging: JSON logs that are machine-parseable
  - Distributed tracing propagation: trace IDs passed in headers
  - Cardinality problems: too many unique metric labels crashes Prometheus
  - OpenTelemetry: vendor-neutral standard for all three pillars

21. SECURITY IN SYSTEM DESIGN
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  Protecting data and systems from unauthorized access or damage.
  Security must be designed in from day one — not added later.

▸ Key concepts (Intermediate)
  - Authentication: who are you? (username/password, OAuth, SSO)
  - Authorization: what can you do? (RBAC, ABAC, JWT claims)
  - Encryption in transit: TLS/HTTPS — data encrypted during travel
  - Encryption at rest: data encrypted when stored on disk

▸ Advanced patterns (Advanced)
  - Zero-Trust: trust nothing by default, verify every request
  - mTLS: both client and server prove identity to each other
  - Secrets management: Vault by HashiCorp — never hardcode credentials
  - OWASP Top 10: injection, broken auth, XSS, SSRF — know them all
  - GDPR/SOC2/ISO27001: compliance frameworks for data handling

22. AI-NATIVE SYSTEM DESIGN (2023–2024)
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  Systems that use AI/LLMs (Large Language Models) as a core component.
  Not just "add AI as a feature" — the AI IS the product (ChatGPT, Copilot).

▸ New infrastructure needs (Intermediate)
  - GPU clusters: training and inference require specialized hardware
  - Model serving: serve predictions at low latency at scale
    Tools: Triton Inference Server, TorchServe, vLLM
  - Vector databases: store embeddings for similarity search
    Tools: Pinecone, Weaviate, Qdrant, pgvector (Postgres extension)
  - Embedding pipelines: convert text/images into vectors for search

▸ RAG Architecture (Advanced)
  RAG = Retrieval-Augmented Generation
  Problem: LLMs hallucinate and have stale knowledge
  Solution:
    1. User asks question
    2. System retrieves relevant docs from vector DB
    3. Retrieved context + question sent to LLM
    4. LLM answers using real, up-to-date context

  Challenges:
  - Chunking strategy: how you split docs affects retrieval quality
  - Re-ranking: retrieved results may need to be re-sorted by relevance
  - Context window limits: you can't stuff too much into one LLM call
  - Hybrid search: combine vector search + keyword search (BM25)

23. LLMOps & AI PLATFORM ENGINEERING (2024)
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  MLOps was about managing ML models in production.
  LLMOps is the same idea, but for Large Language Models.
  Running LLMs in production is very different from running normal APIs.

▸ Key challenges (Intermediate)
  - Prompt versioning: prompts are code, they must be versioned and tested
  - Evaluation (evals): how do you measure if an LLM response is good?
  - Token cost management: LLM API calls are expensive at scale
  - Latency: LLM responses are slow (1-30s). Streaming solves the UX problem.
  - Fine-tuning pipelines: customize base model on your domain data

▸ Production architecture (Advanced)
  - Prompt caching: cache identical prompt+response pairs (Anthropic, OpenAI)
  - Semantic caching: cache responses for *similar* prompts (GPTCache)
  - Model routing: send simple queries to cheap models (GPT-3.5),
                   complex ones to expensive (GPT-4, Claude Opus)
  - Guardrails: detect/block harmful content, PII, off-topic queries
  - Observability for LLMs: log every prompt, response, token count, latency
  Tools: LangSmith, Helicone, Weights & Biases, Braintrust

24. AGENTIC SYSTEMS & MULTI-AGENT ARCHITECTURE (2024–2025)
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  AI agents are LLMs that can take actions — browse the web, write code,
  call APIs, remember things — in a loop until the task is done.
  Multi-agent: multiple AI agents working together on sub-tasks.

▸ How it works (Intermediate)
  The ReAct loop: Reason → Act → Observe → Reason → ...
  - LLM reasons about what to do
  - Calls a tool (search, calculator, code executor, API)
  - Observes the result
  - Reasons again until goal is achieved

  Frameworks: LangGraph, AutoGen, CrewAI, Semantic Kernel

▸ System design concerns (Advanced)
  - Reliability: agents can get stuck in infinite loops
  - Tool design: tools must be clear, idempotent, and safe to retry
  - Memory systems:
      Working memory  → in-context (limited by context window)
      Episodic memory → stores recent interactions (vector DB)
      Long-term memory → persistent knowledge store
  - Human-in-the-loop: when should agents pause and ask a human?
  - Sandboxing: code-executing agents must be isolated (Docker, E2B)
  - Cost control: agent loops consume many LLM tokens — set hard limits

25. VECTOR DATABASES & SEMANTIC SEARCH (2023–2025)
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  Traditional search: "find exact keyword"
  Semantic search: "find things with the same MEANING"
  Vector DBs store embeddings (lists of numbers) that represent meaning.

▸ How it works (Intermediate)
  1. Run text through an embedding model → get a vector (e.g., 1536 numbers)
  2. Store vector in vector DB
  3. At query time: embed the query → find nearest vectors (ANN search)
  4. Return the most semantically similar results

  ANN algorithms: HNSW, IVF, FAISS
  Distance metrics: cosine similarity, dot product, Euclidean

▸ Production concerns (Advanced)
  - Index build time and memory trade-offs (HNSW is fast but RAM-heavy)
  - Metadata filtering: vector search + SQL-style filters combined
  - Multimodal embeddings: same vector space for text AND images (CLIP)
  - Dimensionality reduction: compress vectors to save storage (PCA)
  - Managed vs. self-hosted: Pinecone vs. Weaviate vs. Qdrant

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

26. DATA STREAMING & REAL-TIME PIPELINES (2020–2025)
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  Process data as it arrives — not in batches that run overnight.
  Think: fraud detection at the moment of transaction, not next morning.

▸ Tools (Intermediate)
  Apache Kafka    → distributed event log, handles millions of events/sec
  Apache Flink    → stateful stream processing with exactly-once guarantees
  Apache Spark    → batch + micro-batch streaming (Spark Structured Streaming)
  Kinesis         → AWS managed streaming (simpler than Kafka)

▸ Pipeline patterns (Advanced)
  - Lambda architecture: separate batch layer + speed layer + serving layer
  - Kappa architecture: one streaming layer replaces batch + speed (simpler)
  - Watermarking: handle late-arriving data in windowed aggregations
  - Exactly-once semantics: Kafka transactions + Flink checkpointing
  - Schema registry: enforce Avro/Protobuf schemas on events (Confluent)

28. WEBASSEMBLY (WASM) & PORTABLE COMPUTE (2023–2026)
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  WASM = a binary format that runs code at near-native speed in ANY environment.
  Write in Rust/C/Go → compile to WASM → run in browser, edge, or server.

▸ Use cases (Intermediate)
  - Run complex logic in browser without JavaScript performance limits
  - Edge computing: run business logic at CDN edge nodes
  - Plugin systems: safely run untrusted user code (sandboxed)
  - WASI: WASM System Interface — run WASM outside the browser (like a container)

▸ Why this matters (Advanced)
  - Docker creator Solomon Hykes: "If WASM+WASI existed in 2008, we wouldn't need Docker"
  - WASM components: compose programs from smaller WASM modules
  - Faster cold starts than containers for serverless functions
  - WasmEdge: WASM runtime optimized for cloud and edge

29. MULTI-REGION & GLOBALLY DISTRIBUTED SYSTEMS (2022–2026)
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  Running your system in multiple geographic regions simultaneously.
  Users in India are served by Asia region. Users in Germany by EU region.

▸ Patterns (Intermediate)
  - Active-Passive: primary region handles traffic, secondary on standby
  - Active-Active: all regions serve traffic, data syncs between them
  - Read replicas per region: writes go to primary, reads from local replica
  - Data residency: some data must stay in specific countries (GDPR)

▸ Hard problems (Advanced)
  - Clock skew: machines in different regions have slightly different times
    Solution: logical clocks (Lamport clocks), TrueTime (Google Spanner)
  - Conflict resolution in Active-Active: two regions update same record
    Solution: CRDTs (Conflict-free Replicated Data Types), last-write-wins
  - Global traffic management: Route53 Latency routing, Anycast
  - Disaster recovery: RTO (Recovery Time) vs. RPO (Recovery Point) targets

31. FinOps & COST-AWARE ARCHITECTURE (2023–2026)
────────────────────────────────────────────────────────────────────────────────
▸ What it is (Beginner)
  Cloud is expensive. FinOps = financial operations for cloud.
  Engineers must design with cost in mind, not just performance.

▸ Practices (Intermediate)
  - Right-sizing: don't over-provision compute
  - Spot/Preemptible instances: 70-90% cheaper, but can be killed anytime
  - Reserved instances: commit 1-3 years, get big discount
  - Cost allocation tags: track spending per team/product/feature
  - S3 lifecycle rules: automatically move old data to cheaper storage tiers

▸ Architecture for cost (Advanced)
  - Tiered storage: hot (SSD) → warm (HDD) → cold (Glacier/Archive)
  - Batch vs. real-time: batch is MUCH cheaper; use real-time only when needed
  - LLM cost: the new largest cloud bill line item for AI companies
    Solution: model routing, prompt caching, smaller fine-tuned models
  - Shared tenancy vs dedicated: multi-tenant saves money but adds complexity

KEY MENTAL MODELS EVERY ENGINEER MUST KNOW
──────────────────────────────────────────
  1. CAP Theorem          → You can't have C + A + P all at once
  2. Fallacies of dist.   → Network is NOT reliable, NOT zero-latency
  3. Little's Law         → L = λW (throughput, latency, concurrency)
  4. Amdahl's Law         → Parallelism has diminishing returns
  5. Thundering herd      → Many clients retry simultaneously = worse failure
  6. Backpressure         → Tell producers to slow down when overwhelmed
  7. Idempotency          → Safe to retry — same result every time
  8. Bulkhead pattern     → Isolate failures so one part doesn't sink the whole
  9. Circuit breaker      → Stop calling a failing service; fail fast
 10. Two generals problem → You can never be 100% sure remote received message
